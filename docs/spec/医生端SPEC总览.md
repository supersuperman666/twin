# 医生端SPEC总览

版本：V0.1  
适用端：医生 PC 管理端  
文档定位：承接主 PRD 中医生端页面、交互、字段和开发规则

## 1. 医生端SPEC怎么用

医生端 SPEC 不重复解释完整业务背景，默认业务闭环以 [医患闭环主PRD](../prd/医患闭环主PRD.md) 为准。

这份文档更关注：

- 医生端页面怎么拆
- 每页承担什么职责
- 页面之间如何跳转
- 哪些模块需要单独出深度 SPEC

## 2. 医生端页面主线

当前建议按以下页面组维护医生端 SPEC：

1. 工作台 / 首页
2. 患者管理
3. 患者详情
4. 管理方案
5. 预警处理
6. 随访管理
7. 医生建议与结构化沟通
8. 设备与报告
9. 质控与运营看板

## 3. 各页面职责

### 3.1 工作台 / 首页

目标：

- 让医生快速看到待处理事项，而不是完整报表。

核心内容：

- 待处理预警
- 待确认方案
- 今日随访
- 关键患者提醒

### 3.2 患者管理

目标：

- 高效筛选需要处理的患者。

核心内容：

- 患者搜索
- 疾病标签、风险分、方案状态、待办状态
- 快速进入详情、方案、预警和随访

### 3.3 患者详情

目标：

- 聚合患者档案、趋势、报告、方案、预警、随访和历史处置。

核心内容：

- 患者档案
- 风险标签和医生确认疾病
- 关键趋势
- 当前方案
- 历史随访
- 时间轴

关联深度 SPEC：

- [健康档案页SPEC](./健康档案页SPEC.md)

### 3.4 管理方案

目标：

- 承接方案创建、审核、下发、调整和历史版本查看。

核心内容：

- 方案列表
- 方案详情编辑
- 模块配置
- 患者可见内容预览
- 发布前统一校验
- 底部固定操作条

关联深度 SPEC：

- [医生PC端-管理方案SPEC](./医生PC端-管理方案SPEC.md)

当前页面原则：

- 方案基础信息优先内联编辑，不强制再点“编辑”进入二级态。
- 发布校验统一放在“确认下发”时触发，不在页面顶部和右侧重复轰炸。
- 底部固定操作条保留 `保存草稿 / 确认下发`。
- 方案调整按“原任务关闭 + 新任务生成”的思路承接。

### 3.5 预警处理

目标：

- 让医生快速识别高优先级风险并给出处置。

核心内容：

- 预警列表
- 证据链
- 处理动作：复测、随访、调整方案、转诊建议、忽略并留痕

当前页面原则：

- 预警列表优先突出等级、来源、关键证据和待处理动作。
- 不做复杂大屏，不做批量处理作为 P0 核心能力。
- 与方案、随访之间必须有直接跳转入口。

### 3.6 随访管理

目标：

- 管理随访计划、执行状态和随访结论。

核心内容：

- 随访列表
- 新建/改期
- 随访前准备
- 随访记录
- 复测指标、创建随访、发送医生建议与必要时方案调整入口

当前页面原则：

- 列表页优先按“今日待处理 / 逾期 / 待患者准备”组织。
- 随访详情页以“摘要 + 结构化记录 + 下一步动作”三段组织。
- 不先做复杂排班和 IM，会更稳。

### 3.7 医生建议与结构化沟通

目标：

- 让医生发起可执行、可追踪、结构化的沟通。

核心内容：

- 医生建议列表
- 结构化问题模板
- 患者回复摘要
- 沟通结果回流

### 3.8 设备与报告

目标：

- 查看设备同步状态、睡眠/血氧报告、设备异常。

核心内容：

- 设备绑定状态
- 最新同步时间
- 异常设备列表
- 报告详情

### 3.9 质控与运营看板

目标：

- 面向负责人和管理员查看执行质量，而不是做临床处理。

核心内容：

- 方案下发率
- 任务完成率
- 预警处理时效
- 随访完成率

## 4. 医生端建议优先拆深的 SPEC

优先级建议：

### P0

- 管理方案
- 患者管理
- 患者详情
- 预警处理
- 随访管理

建议下一批新增子 SPEC：

- `医生PC端-预警处理SPEC.md`
- `医生PC端-随访管理SPEC.md`
- `医生PC端-患者管理SPEC.md`

### P1

- 医生建议与结构化沟通
- 设备与报告
- 质控看板

## 5. 医生端与主 PRD 的边界

应该写在主 PRD 的内容：

- 方案调整后旧任务怎么处理
- 预警如何触发复测或随访
- 患者端如何承接任务
- 医患两端状态流转

应该写在医生端 SPEC 的内容：

- 页面布局
- 组件形态
- 字段和枚举
- 按钮和弹层
- 发布校验和交互顺序
