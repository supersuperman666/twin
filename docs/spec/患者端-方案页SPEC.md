# 患者端-方案页SPEC

> 来源：主 PRD《管理方案闭环PRD》《执行任务闭环PRD》、医生端《管理方案SPEC》《干预动作SPEC》与现有小程序产品结构。本文档聚焦患者端底部 `方案` Tab 的页面定位、布局、展示内容和交互闭环。

## 1. 本期目标

P0 方案页必须跑通：

```text
方案页
  -> 查看当前生效中的主方案
  -> 看到最近医生安排
  -> 查看随访安排与准备项
  -> 进入复测、建议、随访准备的具体动作
  -> 在待确认态完成“已知晓”
```

## 2. 页面定位

方案页不是方案列表页，也不是今日任务页，而是：

**当前主方案详情页 + 最近安排页**

它主要解决三个问题：

1. 我现在处于什么管理阶段  
2. 医生最近又安排了什么  
3. 下次随访前我需要准备什么  

与其他 Tab 的分工建议：

- `首页`：今天要做什么，快速执行入口
- `方案`：当前方案、最近安排、随访准备
- `记录`：主动补录与历史查看

## 3. 页面结构

方案页建议做成一个连续详情页，不做卡片列表，不做一级子 Tab。

从上到下布局建议：

1. 当前方案头卡
2. 最近安排
3. 方案明细
4. 随访安排
5. 今日任务轻入口
6. 阶段说明
7. 历史方案弱入口（可选）

## 4. 当前方案头卡

### 4.1 展示内容

- 当前方案名称
- 适用疾病
- 当前阶段目标
- 方案周期
- 生效时间
- 主责医生

### 4.2 页面表达建议

```text
糖尿病管理方案
控糖巩固期

主责医生：林医生
适用疾病：糖尿病
已生效：2026-05-20
周期：30天

本阶段目标：
稳定空腹血糖，减少波动，保持规律记录
```

### 4.3 交互原则

- 直接展示，不需要点击展开才能看核心信息
- 不展示复杂规则阈值
- 不展示方案版本 diff
- 不展示医生内部备注

## 5. 待确认态

医生下发新方案后，在患者确认前，方案页首屏进入待确认态。

### 5.1 展示内容

- 主责医生信息
- 方案名称
- 适用疾病
- 方案周期
- 计划生效时间
- 当前阶段目标
- 方案明细
  - 指标测量
  - 生活方式
  - 用药 / 设备要求（如有）
  - 随访安排

### 5.2 操作

- `我已知晓，开始执行`
- `我有疑问`

交互要求：

- 两个操作按钮固定吸附在页面底部
- 主按钮：`我已知晓，开始执行`
- 次按钮：`我有疑问`
- 页面内容区需预留底部安全间距，避免最后一段方案明细被按钮遮挡

### 5.3 交互原则

- 待确认态必须把方案内容展示得比执行态更完整，帮助患者理解自己将接受什么方案
- 点击 `我已知晓，开始执行` 后，方案状态转入执行中
- 点击 `我有疑问` 后，弹出轻量文本反馈框
- 当前不提供“无法执行”作为通用操作

`我有疑问` 弹层建议：

- 标题：`填写你的问题`
- 内容：单个多行文本框
- 按钮：
  - `取消`
  - `提交`

交互规则：

- 文本框支持用户自由输入问题
- 不做结构化问答
- 提交后提示：`已提交给医生`
- 提交内容进入医生侧患者沟通/方案反馈记录

### 5.4 待确认态页面结构建议

待确认态建议按以下顺序展示：

1. 当前方案头卡
2. 当前阶段目标
3. 方案明细
4. 底部确认操作

设计原则：

- 在确认前，优先让患者看清“长期要做什么”
- 方案明细默认展开，不做深层折叠
- 确认按钮固定在底部，降低漏确认风险

## 6. 执行态首屏

患者确认方案后，方案页首屏切换为执行态。

执行态首屏不再优先展示完整方案内容，而优先展示与当前执行最相关的动态内容。页面形式建议为**连续信息流**，不做页内子 Tab，不做锚点导航。

### 6.1 首屏结构

从上到下建议展示：

1. 当前方案头卡
2. 最近安排
3. 随访安排
4. 今日任务轻入口

方案明细、阶段说明和历史入口下沉到页面中下部，供患者按需回看。

### 6.2 当前方案头卡

执行态下头卡仍然保留，但信息比待确认态更聚焦。

建议展示：

- 当前方案名称
- 当前阶段名称 / 阶段目标摘要
- 适用疾病
- 主责医生
- 生效时间
- 方案周期
- 当前状态：执行中

建议示例：

```text
糖尿病管理方案
控糖巩固期

主责医生：林医生
已生效：2026-05-20
周期：30天

本阶段目标：
稳定空腹血糖，减少波动
```

交互原则：

- 头卡默认直接展示，不折叠
- 不展示版本号、后台来源、内部备注
- 不在头卡内堆叠多个按钮，避免首屏过重

### 6.3 最近安排

这是执行态首屏最重要的动态区块，用于承接医生最近新增且仍有效的动作。

建议展示：

- 最近 1-3 条有效安排
- 按时间倒序
- 每条只保留一个主动作按钮

可展示类型：

- 医生建议
- 复测指标
- 新建随访

展示内容建议：

#### 医生建议卡片

- 标题
- 正文摘要 1-2 行
- 医生姓名
- 发送时间
- 状态：未读 / 已读
- 按钮：`查看`

#### 复测指标卡片

- 标题：`复测指标`
- 指标摘要
- 时长摘要，例如 `连续 3 天`
- 状态：待完成 / 今日已完成 / 已完成 / 已到期
- 按钮：`去记录`

#### 随访安排卡片

- 标题：`随访安排`
- 时间
- 方式：电话 / 线下
- 患者说明摘要
- 状态：待准备 / 已部分完成 / 已准备
- 按钮：`查看准备项`

交互原则：

- 最近安排区只展示“最近新增、当前仍有效”的内容
- 不展示完整历史列表
- 点击卡片主按钮进入对应详情或执行页

### 6.4 随访安排

执行态首屏中，随访安排需要作为独立区块展示，不与最近安排完全混在一起。

首屏只展示**当前最近一次仍有效、最需要患者处理的随访**，不平铺所有随访。

建议展示：

- 最近一次有效随访时间
- 倒计时
- 随访方式
- 本次随访前需要准备的内容摘要
- 当前准备状态

若存在特殊准备项，还应直接显示入口：

- 检查报告：`去上传`
- 量表问卷：`去填写`

交互原则：

- 若最近安排中已有“新建随访”卡片，首屏随访安排区仍保留，因为它承载的是稳定的随访信息
- 点击后进入随访准备详情页
- 普通准备项只作为说明，不全部任务化
- 若存在多个随访，首屏只展示一条，优先级建议为：
  1. 状态为待准备 / 待完成的随访
  2. 计划开始时间最早、最临近当前需要处理的随访
  3. 若同一天有多条，再优先展示创建时间更早的一条

### 6.4.1 多个随访时的展示规则

患者端可能同时存在：

- 当前待准备随访
- 已完成的历史随访
- 已取消或已改期的旧随访

本期不建议在执行态首屏平铺多张随访卡，避免页面过重。

建议规则：

- 首屏只保留 1 张 `当前随访安排` 卡片
- 若还有其他随访记录，在卡片底部展示弱入口：
  - `查看全部随访`
- 若当前没有待准备随访，但存在历史随访，则首屏可展示：
  - `当前暂无待准备随访`
  - `查看历史随访`

### 6.4.2 历史随访入口

历史随访不建议混在首屏信息流里，应进入独立二级页查看。

入口建议放在：

- 执行态首屏 `随访安排` 卡片底部：`查看全部随访`
- 方案页中部 `随访安排` 区块底部：`查看全部随访`

### 6.4.3 全部随访 / 历史随访页

建议进入独立二级页查看全部随访记录。

本期该页建议命名为：

- `随访记录`

页面定位：

- 统一查看患者全部随访
- 不单独再做“随访详情页”
- 每条随访在列表页内直接展开明细和操作

#### 6.4.3.1 页面结构

页面顶部：

- 页面标题：`随访记录`

页面内容区：

- 分组一：`待处理`
- 分组二：`已完成`

每个分组内部采用**卡片时间轴**形式展示，不做普通平铺列表。

本期不再细分更多状态 tab，不做复杂筛选。

#### 6.4.3.2 分组规则

`待处理` 包含：

- 待准备
- 已部分完成
- 已准备

`已完成` 包含：

- 已完成
- 已取消
- 已改期

展示顺序建议：

- `待处理` 分组排在前
- 两个分组内部都按**随访开始时间距离当前时间从最近到最远**排序
- 也就是优先展示“离今天最近、最值得患者先处理或先回看的随访”

#### 6.4.3.2A 时间轴展示形式

建议采用纵向时间轴卡片：

- 左侧：时间轴线 + 时间节点
- 右侧：随访卡片内容

时间轴节点建议：

- `待处理`：使用主色/橙色节点，强调仍需处理
- `已完成`：使用灰绿或浅灰节点，强调已结束

页面阅读顺序：

- 从上到下浏览
- 越靠上，说明离当前时间越近

这样患者能很直观地理解：

- 最近需要处理哪次随访
- 最近一次完成的随访是什么时候
- 更早之前还有哪些随访记录

#### 6.4.3.3 每条随访卡的收起态

收起态用于快速浏览，不展示全部细节。

建议展示：

- 随访开始时间
- 随访类型 / 主题
- 状态标签
- 随访方式
- 一句话摘要

示例：

```text
2026-05-28 10:30
预警后随访        待准备
电话随访
请在随访前补齐近 3 天血糖记录
```

#### 6.4.3.4 每条随访卡的展开态

不做单独详情页，点击卡片后直接在当前列表中展开。

展开后建议展示：

1. 基础信息
   - 随访类型 / 主题
   - 状态
   - 随访开始时间
   - 随访方式

2. 患者说明
   - 本次随访前需要做什么
   - 医生想重点了解什么

3. 准备项清单
   - 普通准备项说明
   - 特殊准备项入口

4. 当前进度
   - 未开始 / 已部分完成 / 已准备 / 已完成

5. 操作区
   - 根据状态展示可操作按钮

#### 6.4.3.5 待处理随访展示内容

待处理随访展开后展示：

- 随访开始时间
- 随访方式
- 患者说明
- 准备项清单
- 当前准备状态
- 特殊准备项入口

准备项展示规则：

- 普通准备项仅做说明展示，不自动任务化
- 检查报告显示：`去上传`
- 量表问卷显示：`去填写`

待处理随访支持的操作：

- `去上传检查报告`
- `去填写量表问卷`
- `收起`

若没有特殊准备项，则不显示操作按钮，仅展示说明和状态。

#### 6.4.3.6 已完成随访展示内容

已完成随访展开后展示：

- 随访开始时间
- 随访方式
- 本次随访主题
- 医生结果摘要
- 后续建议摘要

示例：

```text
常规随访        已完成
2026-05-10 15:00
电话随访

本次结果：
近期血糖较前稳定，建议继续当前方案

后续建议：
继续完成空腹血糖和睡前血压记录
```

已完成随访支持的操作：

- `收起`

本期不要求在已完成随访卡中继续提供二次跳转。

#### 6.4.3.7 已取消 / 已改期展示内容

如该随访处于已取消或已改期，展开后展示：

- 原计划开始时间
- 当前状态
- 简短说明

例如：

- 本次随访已改期，请以最新安排为准
- 本次随访已取消，医生将根据情况重新安排

支持的操作：

- `收起`

#### 6.4.3.8 交互规则

- 页面内不再进入单独随访详情页
- 默认全部卡片收起
- 距离当前时间最近的一张随访卡默认展开
- 同一时间只展开一张卡，再展开另一张时，前一张自动收起
- 报告上传和量表填写直接从展开卡片进入对应页面

#### 6.4.3.9 空状态

若 `待处理` 为空：

- 展示：`当前没有需要准备的随访`

若 `已完成` 为空：

- 展示：`还没有历史随访记录`

### 6.5 今日任务轻入口

执行态首屏可以保留与任务的关系，但不再重复首页的大任务列表。

建议展示：

- 今日任务数量，例如 `今日有 2 项待完成`
- 当前完成进度，例如 `已完成 1/2`
- 最近一条关键任务摘要（可选）
- 主按钮：`去今日任务`

展示示例：

```text
今日任务
今日有 2 项待完成，已完成 1 项

[去今日任务]
```

交互原则：

- 这里只做轻入口，不平铺完整任务列表
- 点击后优先跳转到首页今日任务区或具体任务页
- 若当天无任务，展示：`今天的任务已完成`

### 6.6 下沉内容

以下内容不进入执行态首屏主体，但仍需保留在页面下方：

- 方案明细
- 阶段说明
- 历史方案入口

设计原则：

- 首屏解决“现在最相关的内容是什么”
- 下沉区解决“这套长期方案本身包含什么”

## 7. 最近安排

这是方案页最重要的动态内容区，用来承接医生最近发起的动作。

### 6.1 承接内容

`最近安排` 区承接医生端已定义的患者可见干预动作，动作本身的创建规则、字段与患者侧闭环以 [医生PC端-干预动作SPEC](/Users/ks-hz/Desktop/数字孪生/docs/spec/医生PC端-干预动作SPEC.md) 为准。

本页只负责展示最近仍有效的动作结果，P0 承接：

- 医生建议
- 复测指标
- 新建随访

按时间倒序展示最近 1-3 条即可，不在本页重复定义动作生成规则。

补充说明：

- `最近安排` 只展示当前仍有效、与当前方案执行直接相关的内容
- 历史医生建议不在方案页沉淀，统一进入 `我的 -> 医生建议`
- 历史任务不在方案页沉淀，统一进入 `我的 -> 我的任务`

### 6.2 区块标题建议

`最近安排`

### 6.3 医生建议卡片

展示内容：

- 标题
- 正文摘要（1-2 行）
- 发送医生
- 发送时间
- 状态：未读 / 已读
- 按钮：`查看`

点击后：

- 进入建议详情层
- 首次查看自动变为 `已读`

### 6.4 复测指标卡片

复测指标在患者端建议展示为一个任务组卡片，而不是拆成很多散任务。

展示内容：

- 标题：`复测指标`
- 指标摘要：例如 `血糖（空腹）/ 血压（晨起、睡前）`
- 说明：例如 `连续 3 天完成`
- 状态：待完成 / 今日已完成 / 已完成 / 已到期
- 按钮：`去记录`

若为多指标复测：

- 卡片摘要展示主指标
- 点击后可在二级层选择具体指标去记录

### 6.5 随访安排卡片

展示内容：

- 标题：`随访安排`
- 时间
- 方式：电话 / 线下
- 说明：请在随访前准备以下内容
- 状态：待准备 / 已部分完成 / 已准备
- 按钮：`查看准备项`

### 6.6 交互原则

- 最近安排区优先展示“最近新增、当前仍有效”的动作
- 不做完整历史列表
- 每条卡片都有明确单一主动作

## 8. 方案明细

方案明细用于承接当前主方案本身包含的长期内容，不等同于“最近新增动作”。

### 8.1 展示分组

- 指标测量
- 生活方式
- 用药 / 设备
- 随访节奏

### 8.2 指标测量

展示患者长期需要关注和记录的指标，不直接暴露后台配置字段。

患者端不直接展示后台原始配置，例如：

- 指标名称
- 测量场景
- 执行频率
- 方案周期

而是由系统根据后台配置自动编译成患者容易理解的文案。

#### 8.2.1 编译目标

患者看到的不是“规则”，而是：

1. 这段时间需要重点关注什么指标
2. 大概在什么场景下关注
3. 是持续关注，还是按需补充

#### 8.2.2 编译规则

系统根据 `指标名称 + 测量场景 + 执行频率` 生成患者端展示文案。

基础规则：

- 不展示后台字段名，如“执行频率”“multi-select”“time range”
- 不直接暴露复杂规则表达，如“每日2次、连续30天”
- 优先翻译成“长期关注要求”，避免让方案明细看起来像任务清单
- 相同指标的多个场景合并表达，不拆成很多行

| 睡眠报告 | 睡眠情况 |

#### 8.2.5 频率映射

频率不能被过度抽象，患者端方案明细需要较真实地表达医生要求，只是改写成患者容易理解的说法。

建议映射如下：

| 后台频率 | 患者端表达 |
| --- | --- |
| 每日1次 | 每天测 1 次 |
| 每日2次 | 每天测 2 次 |
| 每日3次 | 每天测 3 次 |
| 每日4次 | 每天测 4 次 |
| 每 8 小时 1 次 | 大约每 8 小时测 1 次 |
| 每 12 小时 1 次 | 大约每 12 小时测 1 次 |
| 隔日1次 | 每隔一天测 1 次 |
| 每3日1次 | 每 3 天测 1 次 |
| 每周 N 天 | 每周按要求测 N 天 |
| 一次 | 按要求完成 1 次 |
| 必要时 | 不舒服或医生提醒时补充记录 |
| 紧急时 | 出现明显异常时立即记录或补充 |

说明：

- 患者端可以展示真实频率要求
- 但仍不直接展示后台字段名，如 `frequency selector`
- 若同一指标配置了多个场景，频率应和场景一起读起来通顺，例如：`空腹血糖：每天测 1 次`
- 若患者当天需要执行的具体任务已生成，首页今日任务和复测指标仍是主执行入口；方案明细负责说明“长期要求是什么”

#### 8.2.6 文案生成模板

建议按以下模板编译：

```text

```

- 空腹血糖：每天测 1 次
- 餐后血糖：每天测 1 次
- 晨起血压：每周按要求测 3 天
- 睡前血氧：每天测 1 次
- 睡眠情况：每周按要求补充 3 次

#### 8.2.9 前端展示要求

- 指标测量区内容必须按医生端已配置的指标项逐条编译展示，不额外写死固定文案
- 每条指标文案最多展示两行
- 默认先展示 3 条，超出后通过 `展开更多` 查看
- 不在方案明细中展示：
  - 目标范围数值
  - 专业分析指标（如 AHI、ODI、最低血氧）
- 若患者需要具体执行内容，点击相关项后跳转到：
  - 首页今日任务
  - 记录页对应指标
  - 睡眠报告页

### 8.3 生活方式

生活方式不能展示成平台固定文案，必须根据医生端生活方式方案配置自动编译。

#### 8.3.1 编译目标

患者端展示的是：

- 这一阶段医生希望我长期注意什么
- 饮食、运动、作息方面要怎么配合

#### 8.3.2 编译规则

- 按医生端已启用的生活方式项逐条展示
- 有明确频率/次数要求时，保留真实要求
- 没有明确频率时，展示为持续性建议
- 不展示后台字段名，如“生活方式模块”“执行配置”

#### 8.3.3 示例

后台配置：

- 饮食：控制晚餐主食量
- 运动：每周运动 3 次
- 作息：23:00 前入睡

患者端展示：

```text
生活方式
- 晚餐注意控制主食量
- 每周运动 3 次
- 尽量在 23:00 前休息
```

#### 8.3.4 前端展示要求

- 生活方式区内容按医生配置项展示，不做固定模板兜底
- 默认展示前 3 条，超出后可展开
- 每条建议以短句展示，不写成长段文章

### 8.4 用药 / 设备

用药 / 设备区只在医生端明确启用相关方案时展示，并按配置结果自动编译。

#### 8.4.1 用药展示规则

- 若医生端启用了用药方案，患者端展示“需要配合记录什么”，不展示处方后台语言
- 重点展示：
  - 是否需要记录用药执行
  - 是否需要记录漏服 / 不适
  - 是否有固定提醒频次

示例：

```text
用药配合
- 请按医生要求记录每日用药情况
- 如漏服或服药后不适，请及时补充说明
```

#### 8.4.2 设备展示规则

- 若医生端启用了设备监测方案，患者端展示具体设备相关要求
- 重点展示：
  - 是否需要保持设备同步
  - 是否需要使用睡眠监测设备
  - 是否需要补充上传设备数据

示例：

```text
设备配合
- 请保持设备正常同步
- 睡眠监测设备请按医生要求使用
```

#### 8.4.3 前端展示要求

- 用药与设备内容根据医生端是否启用模块决定是否出现
- 若只启用其一，则只展示对应分组
- 不展示设备专业参数，不展示后台模块名

### 8.5 随访节奏

随访节奏不写死为固定说明，必须根据医生端随访计划配置自动编译。

#### 8.5.1 编译目标

让患者知道：

- 这一阶段会不会有持续随访
- 大概需要多久复盘一次
- 目前最近一次随访是什么时候

#### 8.5.2 编译规则

- 若医生端仅配置了下一次随访时间，则展示最近一次有效安排
- 若医生端配置了阶段性随访节奏，则同时展示长期节奏说明
- 不展示后台字段名，如“计划内随访”“随访频率配置”

#### 8.5.3 示例

后台配置：

- 方案周期：30 天
- 随访计划：2 周后随访 1 次

患者端展示：

```text
随访安排
- 本阶段医生将按计划进行随访复盘
- 下次随访时间：2 周后
```

后台配置：

- 方案周期：14 天
- 随访计划：3 天内预警后随访、1 周后复盘

患者端展示：

```text
随访安排
- 本阶段将根据情况安排随访跟进
- 最近一次随访：1 周后复盘
```

#### 8.5.4 前端展示要求

- 随访节奏区必须根据医生端已配置计划编译，不使用统一固定文案
- 该区负责说明“本阶段如何随访”，具体最近一次随访卡片仍在 `随访安排` 区单独展示

### 8.6 交互原则

- 执行态下方案明细默认展示摘要，支持展开查看更多
- 待确认态下方案明细默认完整展开
- 不使用医生端字段名、规则名、阈值语义
- 方案明细负责解释“长期底盘”，最近安排负责承接“最新动作”

## 9. 随访安排

该区块承接当前最近一次有效随访。

### 7.1 展示内容

- 下次随访时间
- 倒计时
- 随访方式
- 患者端说明
- 准备项摘要

### 7.2 准备项展示规则

#### 普通准备项

仅作为说明展示，不自动生成任务：

- 血糖记录
- 血压记录
- 血氧/呼吸数据
- 症状记录
- 用药记录
- 设备同步
- 睡眠报告

展示文案建议：

`本次随访前请准备：近3天血糖记录、血压记录、症状情况`

#### 特殊准备项

提供明确动作入口：

- 检查报告 -> `去上传`
- 量表问卷 -> `去填写`

### 7.3 点击后的交互

点击 `查看准备项` 进入随访准备详情页或抽屉，展示：

1. 随访基础信息
2. 准备项说明
3. 检查报告上传入口（如有）
4. 量表问卷填写入口（如有）

### 7.4 状态规则

- 未完成任何特殊准备项：`待准备`
- 完成部分特殊准备项：`已部分完成`
- 所有特殊准备项完成：`已准备`

普通准备项不单独做完成勾选。

## 10. 今日任务轻入口

首页已经承担“今日任务主入口”，方案页不再重复大块任务列表。

### 8.1 展示方式建议

只做一个轻量入口区：

```text
今日有 2 项待完成
按钮：去今日任务
```

或展示最近一条关键任务：

```text
今日关键任务：复测血糖
按钮：去记录
```

### 8.2 设计原则

- 不在方案页重复首页的大任务流
- 只保留轻量承接，帮助用户从方案页回到执行入口

## 11. 阶段说明

这是方案页的解释层，用于帮助患者理解当前方案在管什么。

### 9.1 展示内容

- 本阶段重点
- 日常持续关注点
- 异常时怎么做

### 9.2 展示原则

- 控制在 3-5 条
- 使用患者能理解的表达
- 不写成长篇教育内容
- 不展示后台规则和专业术语缩写

## 12. 历史方案入口

若本期支持历史方案查看，建议只作为弱入口放在页面底部：

- `查看历史方案`

不建议首屏做方案卡片列表，因为当前患者只有一个生效中的主方案。

### 12.1 历史方案页

点击 `查看历史方案` 后，进入独立二级页：

- 页面名称：`历史方案`

本期建议做成**历史方案卡片页**，不单独再做“历史方案详情页”。

### 12.2 页面定位

用于查看患者此前执行过的方案记录，帮助患者理解：

- 之前执行过哪些阶段方案
- 每个历史方案大概持续了多久
- 当时主要目标是什么

本期不承载复杂版本 diff，不展示医生端内部版本管理信息。

### 12.3 页面结构

建议采用连续卡片列表：

- 按时间倒序展示
- 最近结束的历史方案排在最前

每张历史方案卡片支持：

- 收起态摘要
- 展开态明细

不再额外进入第三层详情页。

### 12.4 收起态展示内容

每张历史方案卡建议展示：

- 方案名称
- 适用疾病
- 生效时间 - 结束时间
- 阶段目标摘要
- 状态标签

状态标签建议：

- 已完成
- 已停用
- 已替换

示例：

```text
糖尿病管理方案
2026-04-01 至 2026-04-30

阶段目标：
稳定空腹血糖，减少波动

[已完成]
```

### 12.5 展开态展示内容

历史方案展开后，直接在卡片内展示方案明细，不再单独做详情页。

展示内容建议：

1. 当前阶段目标
2. 方案明细
   - 指标测量
   - 生活方式
   - 用药 / 设备
   - 随访节奏

### 12.6 方案明细复用规则

历史方案的方案明细展示逻辑与当前方案页保持一致，不另起一套规则。

也就是说：

- 指标测量：继续按医生配置自动编译成患者可理解文案
- 生活方式：继续按医生配置项编译展示
- 用药 / 设备：继续按是否启用和实际配置展示
- 随访节奏：继续按医生端随访计划配置展示

本期不需要为历史方案单独再定义一版明细生成逻辑。

### 12.7 交互规则

- 默认全部历史方案卡片收起
- 一次只展开一张卡
- 点击另一张卡时，前一张自动收起
- 若历史方案较多，支持分页加载或滚动加载，P0 可先用滚动加载 mock 数据

### 12.8 空状态

若没有历史方案：

- 展示：`当前还没有历史方案`

### 12.9 与当前方案页的边界

- 当前方案页：展示正在生效的主方案
- 历史方案页：展示已结束、已停用或已被替换的旧方案

两者不混排在同一页面首屏，避免干扰患者对“当前正在执行什么”的理解。

## 12A. 页面状态清单

为便于前端开发，方案页建议至少覆盖以下页面状态与空状态。

### 12A.1 无方案

适用场景：

- 当前没有生效中的主方案
- 医生草稿未下发
- 历史方案已结束，暂无新方案

页面表现建议：

- 展示空状态卡
- 文案：`当前还没有生效中的管理方案`
- 辅助说明：`医生确认并下发后，你可以在这里查看当前方案和最近安排`

展示规则：

- 不展示方案头卡
- 不展示最近安排
- 不展示方案明细
- 若存在历史方案，可保留弱入口：`查看历史方案`

### 12A.2 待确认

适用场景：

- 医生已下发方案
- 患者尚未点击 `我已知晓，开始执行`

页面表现建议：

- 进入待确认态
- 完整展示：
  - 当前方案头卡
  - 当前阶段目标
  - 方案明细
  - 吸底确认按钮

操作：

- `我已知晓，开始执行`
- `我有疑问`

### 12A.3 执行中

适用场景：

- 患者已确认方案
- 当前存在生效中的主方案

页面表现建议：

- 进入执行态
- 首屏展示：
  - 当前方案头卡
  - 最近安排
  - 随访安排
  - 今日任务轻入口

### 12A.4 无最近安排

适用场景：

- 当前没有医生建议
- 没有进行中的复测指标
- 没有新增随访动作

页面表现建议：

- 保留 `最近安排` 区块标题
- 区块内容显示空状态
- 文案：`当前没有新的安排`

### 12A.5 无随访

适用场景：

- 当前没有待准备随访
- 当前没有新的随访安排

页面表现建议：

- 保留 `随访安排` 区块
- 文案：`当前没有需要准备的随访`
- 若存在历史随访，可展示弱入口：`查看全部随访`

### 12A.6 无今日任务

适用场景：

- 当天没有待完成的计划内任务
- 没有当天复测任务
- 没有当天必须完成的随访准备动作

页面表现建议：

- 保留 `今日任务轻入口` 区块
- 文案：`今天的任务已完成`

### 12A.7 无历史方案

适用场景：

- 患者暂无历史方案

页面表现建议：

- 隐藏 `查看历史方案` 弱入口

### 12A.8 条件模块未启用

适用场景：

- 当前方案存在基础信息
- 但生活方式 / 用药 / 设备等条件模块部分未启用

页面表现建议：

- 只展示已启用且患者可见的方案明细分组
- 不展示空白分组卡

## 15. 设计思路总结

这版方案页设计有三个核心原则：

### 15.1 不做列表

因为当前只有一个生效中的主方案，做列表会让页面空且没有价值。

### 15.2 不和首页重复

首页负责“今天做什么”，方案页负责“这阶段为什么这样做、最近又安排了什么”。

### 15.3 区分待确认态与执行态

- 待确认态重点是“把方案讲清楚”，帮助患者做确认决策
- 执行态重点是“把最近安排和随访准备放前面”，帮助患者继续执行

### 15.4 强化最近安排

在慢病管理里，患者更关心：

- 最近医生有没有新建议
- 现在是否要复测
- 下次随访怎么准备

所以方案页必须强化“最近安排”和“随访准备”，而不是只做静态说明页。

## 16. 一句话总结

患者端 `方案` Tab 最适合做成：**当前主方案详情页 + 最近安排页 + 随访准备页**。  
不做方案列表，不重复首页今日任务；待确认态重点展示完整方案内容，执行态重点承接医生建议、复测指标和随访安排。
