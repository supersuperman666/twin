# 医生PC端-干预动作SPEC

> 文档定位：定义医生 PC 端三类核心干预动作 `发送医生建议 / 复测指标 / 创建随访` 的入口、页面布局、字段、交互和留痕规则。本文档聚焦可开发颗粒度，不回写业务主 PRD。

## 1. 本期范围

P0 只支持以下三类医生动作：

- `发送医生建议`
- `复测指标`
- `创建随访`

不再引入 `临时干预` 这一中间概念，避免医生额外理解一层抽象。

## 2. 设计原则

### 2.1 医生心智优先

医生面对异常或复盘时，最自然的决策不是“我要不要做临时干预”，而是：

- 我要给患者发一条建议
- 我要让患者复测几个指标
- 我要安排一次随访

因此入口和交互必须直接对应动作，不增加概念跳转层。

### 2.2 统一交互原则

三类动作统一遵循：

- 尽量从当前页面上下文带入患者、来源事件和推荐内容
- 尽量使用右侧抽屉，而不是跳新页面
- 尽量提供默认值和推荐项，减少从零填写
- 创建前让医生知道患者端会看到什么
- 创建后不强制跳页，仅局部刷新与留痕

## 3. 统一入口设计

### 3.1 P0 页面入口

| 页面 | 入口按钮 | 位置 | 备注 |
| --- | --- | --- | --- |
| 患者详情页 | 发送医生建议 / 复测指标 / 创建随访 | 顶部患者条 | 面向当前患者的高频快捷操作入口 |
| 预警列表页 | 发送医生建议 / 复测指标 / 创建随访 | 每个预警卡片的操作区 | 支持不进详情页快速发起动作 |
| 预警详情页 | 发送医生建议 / 复测指标 / 创建随访 | 右侧处理区 | 预警后的核心动作区 |
| 随访详情页 | 发送医生建议 / 复测指标 / 创建随访 | 右侧下一步动作区 | 随访复盘后的核心动作区 |
| 患者详情页 | 发送医生建议 / 复测指标 / 创建随访 | 更多操作 / 当前方案卡片 | 医生主动发起 |

### 3.2 不同页面的主次关系

#### 预警详情页

默认主次建议：

1. `复测指标`
2. `创建随访`
3. `发送医生建议`

原因：预警后优先补数据和短期跟进，其次才是纯沟通。

补充规则：

- P0 在预警详情页动作区不直接暴露 `调整管理方案` 入口。
- 若医生判断长期方案确需调整，应从患者当前方案上下文进入，不作为预警处理主动作。

#### 预警列表页卡片

默认主次建议：

1. `复测指标`
2. `创建随访`
3. `发送医生建议`

规则建议：

- 卡片上只展示 2 个主操作 + 1 个更多操作，避免列表过宽
- 默认显示：
  - `复测指标`
  - `创建随访`
- `发送医生建议` 放入更多操作，或在宽屏下作为第三操作展示
- 点击卡片主体仍进入预警详情页

#### 患者详情页顶部患者条

默认主次建议：

1. `发送医生建议`
2. `复测指标`
3. `创建随访`

原因：患者详情顶部患者条更适合承接“对当前患者直接发起动作”，主动沟通优先级更高。

#### 随访详情页

默认主次建议：

1. `创建随访`
2. `复测指标`
3. `发送医生建议`
4. `调整管理方案`

原因：随访后优先安排下一步跟进，其次补观察，建议作为补充动作。

#### 患者详情页

默认主次建议：

1. `发送医生建议`
2. `复测指标`
3. `创建随访`

原因：患者详情页更像单患者综合工作台，主动沟通是更常见入口。

## 4. 三类动作的统一交互框架

### 4.1 统一打开方式

三类动作均使用右侧抽屉。

原因：

- 不打断当前上下文
- 便于保留左侧证据和中间摘要供医生参考
- 更适合少量字段快速确认

### 4.2 统一抽屉骨架

```text
抽屉头部
  标题
  患者信息
  来源事件标签

第一组：对象与来源
  患者 / 来源 / 关联预警 / 关联随访 / 关联方案

第二组：动作内容
  根据不同动作动态展示核心字段

第三组：患者端呈现
  患者说明 / 是否提醒 / 是否生成待办

底部固定操作区
  取消 | 预览患者端内容 | 确认创建/发送
```

### 4.3 统一默认带入

#### 从预警详情进入

自动带入：

- 患者
- 当前预警

补充规则：

- 只带入发起动作所必需的关联对象
- 不默认展开大段预警摘要
- 推荐指标或模板仅作为弱提示，不直接写入表单字段

#### 从预警列表卡片进入

自动带入：

- 患者
- 当前预警

交互要求：

- 从卡片快捷发起时，抽屉头部展示来源预警标签即可
- 抽屉提交成功后，卡片局部刷新动作摘要和处理状态

#### 从随访详情进入

自动带入：

- 患者
- 当前随访

#### 从患者详情页进入

自动带入：

- 患者

#### 从患者详情顶部患者条进入

自动带入：

- 当前患者

交互要求：

- 入口属于全局快捷动作，不强依赖某个单独事件
- 若当前患者没有执行中方案，仍允许发送医生建议和创建随访
- 若没有明确异常上下文，`复测指标` 不自动推荐具体指标，由医生自行选择

### 4.4 创建成功后的统一反馈

- toast 成功提示
- 抽屉关闭
- 原页面局部刷新
- 时间轴新增事件
- 来源页动作区显示已创建记录摘要

不建议创建成功后跳转到新页面。

### 4.4A 无上下文创建规则

从患者详情页直接发起时，即使当前没有：

- 预警
- 随访
- 执行中方案

也允许发起：

- `发送医生建议`
- `复测指标`
- `创建随访`

前端口径：

- `发送医生建议`：直接打开抽屉
- `创建随访`：直接打开抽屉
- `复测指标`：直接打开抽屉，但不自动推荐指标，医生自行选择

### 4.4B 通用按钮与报错规则

- 主按钮在必填项未满足前置灰
- 字段级错误优先展示在字段下方
- 提交失败使用 toast 提示
- 不做顶部全局报错汇总
- 抽屉关闭前若有未保存输入，弹二次确认

### 4.5 与任务模块的分工

- 动作 SPEC 负责说明：医生从哪里发起、填什么、会生成什么、患者端从哪里承接
- 任务模块负责说明：底层任务如何拆分、如何聚合、如何判断完成、如何进入关闭
- 因此 `复测指标` 的任务完成判定规则，不在本文件展开，以 [执行任务闭环PRD](/Users/ks-hz/Desktop/数字孪生/docs/prd/执行任务闭环PRD.md) 为准

## 5. 动作一：发送医生建议

### 5.1 使用场景

- 需要告知患者下一步怎么做
- 需要提醒患者联系医生或线下就医
- 需要把随访结论翻译成患者可理解建议
- 需要配合复测指标或随访，补一条说明

### 5.2 页面入口

| 来源页面 | 入口文案 | 位置 |
| --- | --- | --- |
| 预警列表页 | 发送医生建议 | 每个预警卡片的操作区 / 更多操作 |
| 预警详情页 | 发送医生建议 | 右侧处理区 |
| 随访详情页 | 发送医生建议 | 右侧下一步动作区 |
| 患者详情页 | 发送医生建议 | 更多操作 |

### 5.3 抽屉标题

`发送医生建议`

### 5.4 页面布局

```text
头部
  发送医生建议
  患者姓名 / 来源事件

第一组：来源信息
  来源
  发送原因

第二组：建议内容
  标题
  正文

底部
  取消 | 发送建议
```

### 5.5 字段定义

| 字段 | 类型 | 必填 | 默认/规则 | 说明 |
| --- | --- | --- | --- | --- |
| 来源 | auto | 是 | 预警 / 随访 / 方案 / 手动 | 自动带入，不可编辑 |
| 发送原因 | textarea | 否 | 医生内部留痕 | 不对患者展示 |
| 标题 | input | 是 | 20字内优先 | 患者端卡片标题 |
| 正文 | textarea | 是 | 30-150字建议 | 患者可理解表达 |

### 5.6 交互规则

- P0 采用纯文本建议，不做结构化建议类型和模板体系
- 不支持在发送医生建议时直接配置待办，待办由 `复测指标` 或 `创建随访` 承接
- 内容中禁止出现复杂规则阈值、医生内部推理、专业缩写堆砌

### 5.7 校验规则

- `标题` 必填，不允许纯空格
- `正文` 必填，不允许纯空格
- `标题` 最长 20 字
- `正文` 最长 200 字
- `发送原因` 非必填，但如填写则最长 200 字

### 5.7A 按钮状态与错误提示

- `发送建议` 按钮在 `标题` 或 `正文` 为空时置灰
- `标题` 超长时在字段下提示：`标题不能超过20字`
- `正文` 超长时在字段下提示：`正文不能超过200字`
- 若提交失败，toast 提示：`发送失败，请稍后重试`

### 5.8 创建结果对象

创建后生成 1 条 `doctor_advice` 记录。

建议字段：

- `id`
- `patient_id`
- `source_type`
- `source_id`
- `title`
- `content`
- `reason`
- `status`
- `sent_at`

状态建议：

- `sent`
- `read`
- `expired`

### 5.9 患者端展示

患者端显示：

- 建议标题
- 建议内容
- 来源医生
- 是否已读

展示位置：

- 首页提醒区
- 方案页消息/提醒承接区

### 5.10 留痕要求

时间轴记录：

- 医生发送建议
- 患者已读

### 5.11 创建后的前端回填

#### 从预警列表页发起

- 当前预警卡片显示：`已发送建议`

#### 从预警详情页发起

- 右侧处理区新增一条已发送建议摘要
- 时间轴区域出现新事件

#### 从随访详情页发起

- 右侧下一步动作区下方显示最近一次已发送建议摘要

#### 从患者详情页发起

- 当前患者时间轴新增“发送医生建议”事件

## 6. 动作二：复测指标

### 6.1 使用场景

- 预警后需要短期补数据
- 随访后需要补一轮观察
- 医生主动要求患者未来几天加测指标

### 6.2 页面入口

| 来源页面 | 入口文案 | 位置 |
| --- | --- | --- |
| 预警列表页 | 复测指标 | 每个预警卡片的操作区 / 更多操作 |
| 预警详情页 | 复测指标 | 右侧处理区 |
| 随访详情页 | 复测指标 | 右侧下一步动作区 |
| 患者详情页 | 复测指标 | 更多操作 |

### 6.3 抽屉标题

`创建复测指标`

### 6.4 页面布局

```text
头部
  创建复测指标
  患者姓名 / 来源事件

第一组：复测对象与原因
  关联指标
  复测原因

第二组：执行规则
  测量场景
  执行频率
  复测时长

第三组：患者承接方式
  患者端说明
  是否提醒患者
  开始时间
  截止时间

底部
  取消 | 预览患者端内容 | 创建复测
```

### 6.5 字段定义

| 字段 | 类型 | 必填 | 默认/规则 | 说明 |
| --- | --- | --- | --- | --- |
| 指标名称 | multi-select | 是 | 血糖 / 血压 / SpO2 / 呼吸频率 / 睡眠报告 | 支持多选 |
| 复测原因 | textarea | 是 | 可自动带入预警或随访摘要 | 医生内部留痕 |
| 测量场景 | multi-select | 条件必填 | 按指标动态展示 | 不平铺所有场景 |
| 执行频率 | select | 是 | 一次 / 每日1次 / 每日2次 / 每日3次 | P0 不做复杂规则引擎 |
| 复测时长 | select/input | 是 | 1天 / 2天 / 3天 / 5天 / 7天 / 自定义 | 决定任务周期 |
| 患者端说明 | textarea | 是 | 20-120字 | 不使用专业术语 |
| 是否提醒患者 | switch | 是 | 默认开启 | 开启消息提醒 |
| 开始时间 | date/datetime | 是 | 默认当前或次日 | |
| 截止时间 | date/datetime | 是 | 自动按时长计算，可改 | |

### 6.6 指标与场景枚举

血糖：

- 凌晨
- 空腹
- 早餐后2h
- 午餐前
- 午餐后2h
- 晚餐前
- 睡前
- 随机

血压：

- 晨起
- 睡前
- 随机

SpO2 / 呼吸频率：

- 静息后
- 活动后
- 睡前

睡眠报告：

- 夜间
- 起床后
- 每次睡眠报告后

### 6.7 交互规则

- 先点 `复测指标`，再选指标，不在入口层平铺指标
- 指标选择后动态展示对应的 `测量场景`
- `测量场景` 使用多选下拉，不直接整块平铺
- `执行频率 + 复测时长` 建议同一行展示，方便医生快速确认强度
- 不根据来源事件自动填充推荐组合，由医生自行选择指标、场景、频率和时长

### 6.8 校验规则

- `指标名称` 必填，至少选择 1 个，最多选择 3 个
- `复测原因` 必填，不允许纯空格，最长 200 字
- `执行频率` 必填
- `复测时长` 必填
- `患者端说明` 必填，不允许纯空格，最长 120 字
- `开始时间` 必填，不能早于当前时间
- `截止时间` 非空时不能早于 `开始时间`
- 若未填写 `截止时间`，系统自动根据 `开始时间 + 复测时长` 计算
- `测量场景` 对需要场景的指标条件必填

### 6.8A 按钮状态与错误提示

- `创建复测` 按钮在以下任一条件不满足时置灰：
  - 未选择指标
  - 未填写复测原因
  - 未选择执行频率
  - 未选择复测时长
  - 未填写患者端说明
  - 未填写开始时间
- 场景缺失时，在对应字段下提示：`请选择测量场景`
- 指标超出上限时，toast 提示：`最多选择3个指标`
- 时间冲突时，字段下提示：`截止时间不能早于开始时间`

### 6.9 创建结果对象

创建后生成：

- 1 个 `recheck_plan`
- 底层若干 `patient_task`

建议字段：

- `id`
- `patient_id`
- `source_type`
- `source_id`
- `metric_codes`
- `scenes`
- `frequency`
- `duration_days`
- `start_at`
- `end_at`
- `patient_instruction`
- `status`

状态建议：

- `active`
- `completed`
- `closed`

### 6.10 任务生成规则

- 一次医生发起的 `复测指标`，前台视为 1 个复测动作
- 患者端展示为 1 个复测任务组卡片，不平铺成多条分散任务
- 若选择多个指标，则在同一任务组内展示多个子项
- 若设置 `每日N次 + 连续M天`，底层生成对应执行节点，但患者端按“今日需完成内容”聚合展示
- 任务是否完成的底层判定规则，以 [执行任务闭环PRD](/Users/ks-hz/Desktop/数字孪生/docs/prd/执行任务闭环PRD.md) 为准

### 6.11 患者端承接页面口径

患者端展示为：

- `复测指标`

展示位置：

- 首页今日待办
- 方案页今日任务

患者看到的核心信息：

- 需要复测什么
- 什么时候测
- 连续测几天
- 为什么要测

患者操作：

- `去记录`

### 6.12 留痕要求

时间轴记录：

- 医生创建复测指标
- 患者完成复测
- 复测到期关闭

### 6.13 创建后的前端回填

#### 从预警列表页发起

- 当前预警卡片显示：`已创建复测`
- 卡片状态区可展示简短摘要，如：`3天复测中`

#### 从预警详情页发起

- 右侧处理区新增一条复测摘要
- 时间轴新增“创建复测指标”事件

#### 从随访详情页发起

- 右侧下一步动作区下方显示最近一次复测摘要

#### 从患者详情页发起

- 当前患者时间轴新增“创建复测指标”事件

## 7. 动作三：创建随访

### 7.1 使用场景

- 预警后需要电话或线下跟进
- 随访后需要追加一次复盘
- 医生主动安排新的沟通节点

### 7.2 页面入口

| 来源页面 | 入口文案 | 位置 |
| --- | --- | --- |
| 预警列表页 | 创建随访 | 每个预警卡片的操作区 / 更多操作 |
| 预警详情页 | 创建随访 | 右侧处理区 |
| 随访详情页 | 创建随访 | 右侧下一步动作区 |
| 患者详情页 | 创建随访 | 更多操作 |

### 7.3 抽屉标题

`创建随访`

### 7.4 页面布局

```text
头部
  创建随访
  患者姓名 / 来源事件

第一组：随访基础信息
  随访类型
  关联方案
  关联预警
  创建原因

第二组：随访安排
  计划时间
  随访方式
  负责人

第三组：患者准备
  准备材料
  患者端说明
  是否提醒患者

底部
  取消 | 预览患者端内容 | 创建随访
```

### 7.5 字段定义

| 字段 | 类型 | 必填 | 默认/规则 | 说明 |
| --- | --- | --- | --- | --- |
| 随访类型 | select | 是 | 预警后 / 复盘 / 主动随访 / 追踪随访 | 从来源推断默认值 |
| 关联方案 | selector/auto | 否 | 默认当前执行中方案 | |
| 关联预警 | auto/selector | 条件必填 | 从预警进入时带入 | |
| 创建原因 | textarea | 是 | 自动带入摘要可编辑 | 医生内部留痕 |
| 计划时间 | datetime | 是 | 支持快捷时间：24h / 48h / 3天 / 1周 | |
| 随访方式 | select | 是 | 电话 / 线下 | |
| 负责人 | selector | 是 | 默认当前医生 | |
| 准备材料 | checklist | 否 | 医生按实际需要勾选；可选项见下方枚举 | 默认作为准备项说明，不自动生成任务 |
| 患者端说明 | textarea | 是 | 20-120字 | 告知患者准备什么 |
| 是否提醒患者 | switch | 是 | 默认开启 | |

### 7.6 交互规则

- 从预警进入时自动带入 `关联预警`
- 从随访详情进入时，默认 `随访类型=追踪随访`
- `计划时间` 提供快捷选择，减少医生手填
- `准备材料` 不自动生成独立任务，默认作为患者端随访准备说明展示
- `检查报告`、`量表问卷` 作为特殊准备项，患者端支持上传/填写入口
- 支持 `预览患者端内容`

### 7.7 校验规则

- `创建原因` 必填，不允许纯空格，最长 200 字
- `计划时间` 必填，不能早于当前时间
- `随访方式` 必填，仅支持 `电话 / 线下`
- `患者端说明` 必填，不允许纯空格，最长 120 字
- 从预警入口进入时，`关联预警` 必须带入
- `准备材料` 非必填，不因未选择而阻止创建
- 勾选 `量表问卷` 时，至少选择 1 个量表

### 7.7A 按钮状态与错误提示

- `创建随访` 按钮在以下任一条件不满足时置灰：
  - 未填写创建原因
  - 未设置计划时间
  - 未选择随访方式
  - 未填写患者端说明
- 勾选 `量表问卷` 但未选具体量表时，在字段下提示：`请选择至少一个量表`
- 计划时间早于当前时间时，在字段下提示：`随访时间不能早于当前时间`

### 7.8 创建结果对象

创建后生成 1 条 `followup_plan`。

建议字段：

- `id`
- `patient_id`
- `source_type`
- `source_id`
- `followup_type`
- `related_plan_id`
- `related_alert_id`
- `scheduled_at`
- `method`
- `prepare_items`
- `patient_instruction`
- `status`

状态建议：

- `pending_patient_prepare`
- `pending_doctor`
- `completed`
- `rescheduled`
- `cancelled`

生成规则：

- 普通准备项仅生成随访准备说明，不自动拆成独立任务
- `检查报告` 生成上传入口配置
- `量表问卷` 生成填写入口配置

### 7.9 准备材料规则

#### 7.9.0 准备材料枚举

准备材料可选项统一为：

- 血糖记录
- 血压记录
- 血氧/呼吸数据
- 症状记录
- 用药记录
- 设备同步
- 睡眠报告
- 检查报告
- 量表问卷

#### 7.9.1 默认作为准备项说明

以下内容默认只作为随访前准备说明，不自动生成任务：

- 血糖记录
- 血压记录
- 血氧/呼吸数据
- 症状记录
- 用药记录
- 设备同步
- 睡眠报告

医生勾选后，患者端展示为：

`本次随访前请准备：xxxx`

#### 7.9.2 特殊支持上传/填写

以下两类作为特殊准备项，患者端提供明确入口：

- 检查报告：上传入口
- 量表问卷：填写入口

量表问卷可在勾选后进一步选择量表类型，P0 建议支持：

- CAT
- mMRC
- ESS
- STOP-Bang

前端展示要求：

- `检查报告`：展示为可点击上传项
- `量表问卷`：展示为可点击填写项
- 上传/填写完成后，在随访准备区标记为 `已完成`

### 7.10 患者端承接页面口径

患者端展示为：

- `随访计划`

展示位置：

- 方案页随访计划

- 随访时间

- 查看准备项

- 填写量表问卷

- 患者完成量表问卷（若有）

## 8. 与正式调方案的边界

这三类动作都属于短期干预或短期承接动作，不等于长期策略改变。

只有以下情况才进入 `调整管理方案`：

- 长期目标改变
- 长期测量频率改变
- 长期随访规则改变
- 长期治疗执行要求改变

如果医生从预警或随访入口进入正式调方案，仍需额外填写：

- `调整类型`：预警后调整 / 随访后调整
- `调整原因`

## 9. 患者端交互闭环说明

本次三类动作在患者端不建议做成统一“医生动作中心”，而应分别落到患者最自然的使用路径里：

- 医生建议 -> 消息/提醒
- 复测指标 -> 今日任务
- 创建随访 -> 随访计划与准备清单

这样更符合慢病患者的真实心智，不会让患者被后台动作名称打扰。

### 9.1 医生建议

#### 9.1.1 患者从哪里看到

- 首页提醒区
- 方案页消息/提醒区

#### 9.1.2 卡片展示建议

```text
标题
一句话建议摘要（正文前 1-2 行）
发送医生
发送时间
状态：未读 / 已读
按钮：查看
```

#### 9.1.3 点击后的交互

- 点击卡片或 `查看` 进入建议详情层
- 详情层只展示：
  - 标题
  - 完整正文
  - 医生署名
  - 发送时间
- 第一次进入详情即自动标记为 `已读`

#### 9.1.4 完成后的状态变化

- 首页卡片从 `未读` 变 `已读`
- 若患者返回列表，卡片弱化展示
- 不要求患者再额外点击“确认”

#### 9.1.5 设计建议

- 纯文本建议不应再做复杂回复或表单
- 一屏内读完，不要再跳二级复杂页面
- 若建议内容很短，也可以使用底部抽屉而不是整页详情

### 9.2 复测指标

#### 9.2.1 患者从哪里看到

- 首页今日待办
- 方案页今日任务

#### 9.2.2 卡片展示建议

复测指标在患者端应展示为一个任务组卡片，而不是多条散任务。

```text
标题：复测指标
指标摘要：血糖（空腹）/ 血压（晨起、睡前）
说明：连续 3 天完成
剩余进度：还需完成 2 天
按钮：去记录
状态：待完成 / 今日已完成 / 已完成 / 已到期
```

若是单指标：

- 直接展示该指标和场景

若是多指标：

- 主卡只展示摘要
- 点开后在二级层里展示子项

#### 9.2.3 点击后的交互

- 点击 `去记录` 后，跳转到对应记录页
- 若一个复测计划涉及多个指标：
  - 先进入复测任务详情层
  - 再选择具体指标去记录

#### 9.2.4 记录完成后的反馈

- 返回任务卡后，显示：
  - `今日已完成`
  - 或 `今日完成 1/2 项`
- 若是多天复测：
  - 展示 `还需完成 X 天`
- 所有要求满足后，状态变为 `已完成`

#### 9.2.5 设计建议

- 不要一次性把未来 3 天或 5 天的所有动作全部平铺给患者
- 只展示“今天需要完成什么”
- 优先给明确行动按钮，而不是给很多说明文字

### 9.3 创建随访

#### 9.3.1 患者从哪里看到

- 方案页随访计划
- 首页提醒区

#### 9.3.2 卡片展示建议

```text
标题：随访安排
时间：2026-05-28 10:00
方式：电话 / 线下
说明：请在随访前准备以下内容
按钮：查看准备项
状态：待准备 / 已部分完成 / 已准备
```

#### 9.3.3 点击后的交互

进入“随访准备详情”页或抽屉，展示三部分：

1. 随访基础信息
   - 时间
   - 方式
   - 医生说明

2. 准备项清单
   - 普通准备项只作为说明展示
   - 例如：
     - 血糖记录
     - 血压记录
     - 症状记录
     - 用药记录

3. 特殊入口
   - 检查报告：`去上传`
   - 量表问卷：`去填写`

#### 9.3.4 完成后的状态变化

- 普通准备项不单独做完成勾选
- 检查报告上传后，对应项变为 `已上传`
- 量表问卷填写后，对应项变为 `已完成`
- 若存在已上传/已完成的特殊项，则随访准备状态可显示为：
  - `已部分完成`
  - `已准备`

#### 9.3.5 设计建议

- 随访准备不应堆成任务中心
- 普通准备项更适合“请提前准备”的说明口吻
- 患者真正需要操作的，只有上传报告和填写量表两个入口

### 9.4 三类动作的患者端状态表达建议

| 动作 | 推荐状态表达 |
| --- | --- |
| 医生建议 | 未读 / 已读 |
| 复测指标 | 待完成 / 今日已完成 / 已完成 / 已到期 |
| 创建随访 | 待准备 / 已部分完成 / 已准备 |

### 9.5 患者端交互总原则

- 少概念，多动作
- 少层级，多明确入口
- 少任务轰炸，多聚合展示
- 不让患者理解“预警、干预、规则”等后台概念

## 10. 开发拆分建议

建议按以下顺序开发：

1. 预警详情页接入三类动作入口
2. 发送医生建议抽屉
3. 复测指标抽屉
4. 创建随访抽屉
5. 患者详情页接入三类动作入口
6. 随访详情页接入三类动作入口
7. 患者端承接和时间轴留痕

## 11. 一句话总结

本期医生端干预动作不需要上升成新的抽象模块，直接把 `发送医生建议 / 复测指标 / 创建随访` 三个动作做深、做顺，最符合医生体验，也最适合当前项目阶段。
