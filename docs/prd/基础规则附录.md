# 基础规则附录

版本：V0.1  
文档定位：统一沉淀跨端共用的状态机、任务类型、核心数据对象和关键流转口径，避免医生端、患者端分别维护两套规则

## 1. 为什么需要这份附录

如果状态机、任务类型、数据对象散落在多个 PRD 和 SPEC 里，最容易出现的问题是：

- 医生端写一套状态
- 患者端理解成另一套状态
- 后端再做第三套字段

这份附录的目标，就是把共用规则只维护一份。

## 2. 方案状态总表

| 状态 | 说明 |
| --- | --- |
| `draft` | 草稿 |
| `pending_patient_confirm` | 已下发待患者确认 |
| `active` | 执行中 |
| `stopped` | 已停用 |
| `completed` | 已完成 |
| `rejected` | 已驳回 |

补充：

- 患者未确认、患者有疑问、即将到期、待复盘都作为标签处理
- 不进入方案主状态

## 3. 任务状态总表

| 状态 | 说明 |
| --- | --- |
| `pending` | 待完成 |
| `completed` | 已完成 |
| `closed` | 已关闭 |

补充：

- `overdue` 只是 `pending` 的逾期标签
- 当前不保留 `unable`
- 当前不保留 `invalidated`

## 4. 随访状态总表

| 状态 | 说明 |
| --- | --- |
| `pending_patient_prepare` | 待患者准备 |
| `pending_doctor` | 待医生处理 |
| `overdue` | 已逾期 |
| `completed` | 已完成 |
| `rescheduled` | 已改期 |
| `cancelled` | 已取消 |

补充：

- 随访后的“临时干预”或“正式调方案”作为后续动作留痕，不单独占用随访状态

## 7. 健康风险分规则摘要

### 7.1 分值范围

- 0-100 分
- 分数越高表示近期状态越稳定

### 7.2 风险等级

| 分值 | 风险 |
| ---: | --- |
| 85-100 | 低风险 |
| 70-84 | 中风险 |
| 50-69 | 高风险 |
| 0-49 | 危急/高风险 |

### 7.3 评分维度

| 维度 | 权重建议 |
| --- | ---: |
| 指标状态 | 40 |
| 趋势变化 | 20 |
| 症状状态 | 15 |
| 用药/设备依从性 | 15 |
| 数据完整度 | 10 |

## 9. 关键流转摘要

### 9.1 方案

```text
draft
  -> pending_patient_confirm
  -> active
  -> stopped / completed
```

### 9.2 任务

```text
pending
  -> completed
  -> closed
```

### 9.3 随访

```text
pending_patient_prepare
  -> pending_doctor
  -> completed / overdue / rescheduled / cancelled
```

## 10. 当前统一产品口径

- 方案调整后，旧任务关闭，新任务生成
- 患者端不提供“无法完成”作为通用任务状态
- 预警规则允许全局默认，不强制医生个体化配置
- 患者指导不保留为独立方案模块
- 患者端看到的是任务，不是后台模块结构
